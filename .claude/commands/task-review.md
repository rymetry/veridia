タスク $ARGUMENTS を検収してください。

あなたは**検収者**です。実装者の自己申告(タスクの「検証方法・根拠」節)を信用せず、実ファイルとコマンド再実行で裏取りします。検収者は修正を行いません(指摘して差し戻す)。

## 前提

- 対象は `status: done` のタスク(または完了報告されたタスク)。`not_started` / 着手前のタスクが指定された場合は検収対象外として報告し終了する
- 手順の正本は `AGENTS.md`(作業フロー・変更ルール)と `.claude/commands/run-task.md`。検収基準はこの2つからの逸脱有無である

## 手順

1. **読み込み**: `docs/tasks/` 配下から $ARGUMENTS のファイルを見つけて読む → frontmatterの `plan_ref` が指す計画節 → 必要ならNorth Starの該当§を読む
2. **DoD再検証**: DoDの各項目を「検証方法・根拠」節の記載に沿って**自分で再実行**して確認する
   - テスト・コマンドで検証可能な項目: 実際に再実行し、記載された結果と一致するか確認する(`uv run pytest`、`uv run ruff check .` 等。uv cacheは `UV_CACHE_DIR=${TMPDIR:-/tmp}/uv-cache` を付ける)
   - 再実行不能な項目(実LLM実行・人間レビュー実施・外部サービス依存等): 記録された証跡(evidence、実行ログ、レビュー記録)の実在を確認し、**「証跡確認のみ・再実行未実施」と明示**する
   - DoD項目のチェックボックスが付いているのに根拠記載がない項目は、それ自体を指摘とする
3. **横断チェック**(タスク個別のDoDに書かれていなくても確認する):
   - `uv run pytest` / `uv run ruff check .` / `uv run ruff format --check .` がグリーン
   - schema変更を含む場合: `uv run python scripts/gen_models.py --check` がグリーン
   - `_index.md` が再生成済み(`--check --generated-on <コミット済み生成日>` で確認)
   - frontmatter記帳の整合: `status: done`、「記録」節に記録先リンクまたは「なし」が記入済み
   - 知見記録の妥当性: 記録「なし」の場合、本当に記録すべき知見がなかったか(ハマりどころ・設計判断・gate較正の気づき)を成果物から判断する
   - redaction: `docs/` 配下への変更にsecret / PII / 本番データの生値がない
   - スコープ外変更の混入: 変更ファイル一式(コミット済みなら該当コミットのdiff、未コミットなら `git status` / `git diff`)がタスクのスコープに収まっている
   - 成果物がNorth Star・計画の該当§と矛盾しない(逸脱があるならADRが先行しているか)
4. **検収の深さ**: 通常タスクは要点を絞った検収でよい。論理密度の高いタスク(schema・validator・ID設計・決定性検証・gate配線・ADR系)は深く検証する(境界値・異常系・契約の必須度まで)
5. **判定と記帳**:
   - **合格**: タスクファイル末尾に検収記録を追記する(`## 検収(task-review)` 節: 日付、再実行した検証と結果、証跡確認のみの項目、指摘なしの旨)
   - **不合格(1回目)**: 指摘を「ファイル / 問題 / 期待する状態」の具体的リストとして報告し、タスクの `status` を `in_progress` に戻し、指摘リストを `## 検収(task-review)` 節としてタスクファイルに追記し、`_index.md` を再生成する。修正は行わない(修正は `/run-task $ARGUMENTS` の再実行で行う)
   - **不合格(2回目)**: タスクファイルに既に検収不合格の記録が1件以上ある場合は、`in_progress` に戻さず `status: blocked` にして人間へエスカレーションする(3周目に入らない。指摘の質か実装の前提に問題がある)。理由をタスクファイルに追記し、`_index.md` を再生成する
6. **次のタスク提示**(ループ用): `_index.md` を確認し、`blocked_by` がすべてdoneになっている `not_started` タスクを列挙して報告する(なければ「着手可能タスクなし」と報告)

## ルール

- 検収は保守的に行う。曖昧な充足は不合格側に倒し、判断がつかない項目は人間に確認を求める
- 根拠リンク・再実行結果のない「合格」を書かない
- 検収者は成果物を修正しない。ただし記帳(検収記録の追記、status差し戻し・blocked化、_index再生成)は検収者が行う
- 不合格回数はタスクファイルに追記された検収記録の件数で数える(手順5の2回目分岐の判定根拠)
